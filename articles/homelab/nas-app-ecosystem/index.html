    從 App 安裝到服務編排：構建主權雲的軟件棧

    很多 HomeLab 玩家在安裝 App 時會陷入一個誤區：直接在主機上 apt install 或者隨意 docker run。時間久了，系統會變成一個充滿殘留配置的「垃圾堆」，一旦想遷移或升級，只能選擇重裝。

    真正的 SRE 做法是將 應用邏輯 (Application Logic) 與 數據層 (Data Layer) 完全解耦。

    核心模式：Bind Mount 數據分離

    不要把數據儲存在 Docker 容器的 writable layer 裡。

    標準實作路徑： 應用 (Docker) $ o$ 綁定掛載 (Bind Mount) $ o$ NAS 共享路徑 (NFS/SMB/ZFS Dataset)。

    這樣做的好處是：你的容器變成了「可丟棄」的。無論是升級版本還是更換伺服器，你只需要拉下新的鏡像，掛載原有的數據路徑，服務在 10 秒內恢復，數據零丟失。

    構建你的「私人雲」功能矩陣

    一個完整的主權基礎設施應該涵蓋以下四個維度： 1. 雲端替代：用 Nextcloud 代替 Google Drive，用 Immich 代替 Google Photos，用 Vaultwarden 掌控所有密碼。 2. 網路底層：用 Pi-hole 或 AdGuard Home 在 DNS 層級過濾全家廣告，用 WireGuard 建立安全的遠端回家隧道。 3. 媒體自動化：建立 Plex/Jellyfin $ o$ *Arr suite (Sonarr/Radarr) 的自動化pipeline，實現從發現到觀看的一鍵流轉。 4. 可觀測性：用 Uptime Kuma 監控服務可用性，用 Netdata 或 Grafana 實時追蹤硬體壓力。

    安全邊界：別把你的 HomeLab 裸奔在公網

    很多人的做法是直接在路由器開 Port Forwarding，這簡直是在邀請駭客來逛家。

    SRE 級別的安全鏈條： 公網請求 $ o$ Cloudflare Tunnel / Tailscale $ o$ Reverse Proxy (Nginx Proxy Manager / Traefik) $ o$ SSO 認證 (Authelia / Authentik) $ o$ 目標 App。

    在請求到達 App 之前，必須經過 身份驗證 (SSO)。只有通過 2FA 認證的請求才能進入內網，這才是真正的數據主權。

    資源分配的「裝箱問題」 (Bin Packing)

    為了防止單一 App (例如某個記憶體洩漏的 Java 程序) 把整台伺服器拖垮，必須實施資源限制： - CPU/RAM Limits：在 Docker Compose 中明確定義 deploy.resources.limits。 - ZFS Quota：為每個 App 的數據集設定配額，防止日誌文件無限增長導致整個儲存池爆滿。

    主權雲的魅力不在於你裝了多少 App，而是在於你如何像管理數據中心一樣，精確地掌控每一個 bit 的流向與每一分資源的分配。
