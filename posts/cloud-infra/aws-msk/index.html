    AWS MSK：實作 IAM 角色型身份驗證與零密鑰管理

    在企業級環境中，管理分散在數百個服務中的 Kafka 認證密鑰 (SASL/SCRAM) 是一場維運噩夢。密鑰洩漏、定期更換密鑰導致的服務中斷，以及權限管理碎片化，是每個 SRE 都要面對的痛點。

    為了實現「零密鑰管理」與「精細權限控制」，IAM 角色型驗證 (IAM Role-based Authentication) 是唯一的正確路徑。透過 AWS IAM，我們可以將權限直接綁定到執行個體的 Role，徹底消除密鑰洩漏風險。

    1. 為什麼選擇 IAM 驗證而非 SASL/SCRAM？

    傳統的 SASL/SCRAM 依賴於儲存在 zookeeper 或內建儲存中的用戶名與密碼。而 IAM 驗證則將身份驗證委託給 AWS 的身份管理系統：

    • 零密鑰管理：應用程式不需要在設定檔中寫死密鑰，而是動態獲取臨時 Token。
    • 精細權限控制：可以利用 IAM Policy 精確定義誰能讀取哪個 Topic，誰能管理 Consumer Group。
    • 生命週期同步：當一個 EC2 實例或 Pod 被刪除時，其對應的權限立即失效，無需手動清理密鑰。

    2. IAM Policy 的三維權限拆解

    為了確保「最小權限原則 (Principle of Least Privilege)」，在設定 MSK 的 IAM Policy 時，建議將權限分為三個維度：

    1) 叢集級權限 (Cluster Level)

    允許 Client 端與叢集建立通信。 - kafka-cluster:Connect, kafka-cluster:AlterCluster, kafka-cluster:DescribeCluster

    2) 主題級權限 (Topic Level)

    控制訊息的讀寫權限。 - kafka-cluster:WriteData, kafka-cluster:ReadData, kafka-cluster:DescribeTopic

    3) 分組級權限 (Group Level)

    管理消費者分組的權限，確保只有授權的 Consumer Group 能提交 Offset。 - kafka-cluster:DescribeGroup, kafka-cluster:AlterGroup

    3. 實作中的 SRE 坑洞

    • 連接器兼容性：部分第三方連接器或舊版 Kafka Client 不支援 IAM 驗證，必須更新到支援 aws-msk-iam-auth 插件的版本。
    • 建立時間延遲：MSK 叢集建立約需 15 分鐘。在 CI/CD Pipeline 中，建議使用 Terraform 的 timeout 設定，避免因等待超時而導致部署失敗。
    • 網路隔離：即使有了 IAM 驗證，Security Group 的設定依然是第一道防線。確保只有對應的應用子網能訪問 MSK 的 9098 端口 (IAM 端口)。

    結語：邁向零信任架構

    IAM 驗證不僅僅是一個技術選擇，更是實現雲端零信任架構 (Zero Trust Architecture) 的核心步驟。

    當你將安全重心從「保護密鑰」轉移到「管理身份 (Identity)」時，基礎設施的安全性將從一個脆弱的單點，轉化為一套可審計、可自動化且高度靈活的權限體系。

    關鍵概念總結： - IAM Role-based Auth：利用 AWS IAM 身份實現的 Kafka 驗證機制。 - Zero Key Management：消除靜態密鑰，降低洩漏風險。 - Principle of Least Privilege：透過三維權限拆解實現最小權限授予。 - MSK (Managed Streaming for Apache Kafka)：AWS 的全託管 Kafka 服務。
