    AWS-CLI：多環境帳號切換的極速實踐

    對於管理多個 AWS 環境（Dev/Staging/Prod）的 SRE 而言，頻繁的帳號切換是最高頻的痛點。如果你還在手動修改 ~/.aws/credentials 或者在每個指令後面加上冗長的 --profile 參數，那麼你的工作流中存在著巨大的低效。

    透過 環境變數隔離 與 Shell Function 封裝，我們可以將切換成本從「手動修改配置」降低至「單次指令」。

    三種切換方案深度對比

    在 AWS-CLI 的操作中，有三種主流的身份切換方式，它們在安全性與便捷性之間各有權衡：

    方案 操作方式 適用場景 優缺點
    --profile 參數 aws s3 ls --profile lab 單次指令 $\checkmark$ 最安全 $\times$ 輸入冗長
    環境變數 export AWS_PROFILE=lab 當前 Session $\checkmark$ 快速 $\times$ 容易忘記切換導致誤操作
    Shell Function change_aws_profile lab 全局快速切換 $\checkmark$ 極速 $\times$ 需配置 .zshrc/.bashrc

    最佳實踐：Shell Function 封裝

    為了達到極致的切換速度，我建議在 .zshrc 或 .bashrc 中封裝一個簡單的函數，將環境變數的設定與視覺提示結合在一起。

    實作代碼

    將以下代碼添加到你的 Shell 設定檔中：

    change_aws_profile() {
    if [ -z "$1" ]; then
        echo "❌ Error: Please provide a profile name."
        return 1
    fi
    export AWS_PROFILE=$1
    export AWS_DEFAULT_PROFILE=$1
    echo "🚀 AWS profile switched to: $1"
}

    執行流程

    source ~/.zshrc $\rightarrow$ change_aws_profile production $\rightarrow$ 驗證身份 $\rightarrow$ 開始操作。

    進階安全建議：結合 aws-vault

    雖然 Shell Function 提升了效率，但 .aws/credentials 以明文存儲密鑰依然是安全漏洞。

    我建議搭配 aws-vault 使用。aws-vault 將密鑰存儲於系統金鑰庫 (如 macOS Keychain)，並在執行指令時動態注入臨時憑據。

    最終組合拳：aws-vault exec production -- [your-command]。這實現了「極速切換」與「最高安全」的完美平衡。

    關鍵概念總結： - AWS Profile：定義不同帳號憑據的配置單元。 - Environment Variables：透過 AWS_PROFILE 快速決定 CLI 的操作對象。 - Shell Wrapper：利用函數封裝簡化反覆的操作過程。 - Credential Vault：將敏感密鑰從明文文件移至安全存儲。
