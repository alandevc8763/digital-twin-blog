    破解 CPU 飢餓：解構 Linux CFS 調度器與 Throttling 的真相

    在微服務架構中，最令人困惑的現象莫過於此：監控面板顯示 CPU 使用率僅 30%，但服務的 p99 延遲卻突然飆升。

    對於大多數開發者來說，這像是「幽靈故障」。但對於 SRE 來說，這通常指向一個明確的內核機制：CFS (Completely Fair Scheduler) 的配額限制 (Quota Throttling)。

    01. 什麼是 CFS？從「固定時間片」到「虛擬運行時間」

    早期的調度器使用固定時間片 (Fixed Quanta)，像發牌一樣每人分 10ms。但這在多權重環境下不公平。Linux 引入了 CFS，將 CPU 分配轉化為一場基於權重的競爭。

    核心機制：vruntime (虛擬運行時間)

    CFS 不再關注你「分到了多少時間」，而關注你「跑了多久」。它為每個任務維護一個 vruntime： $$\text{vruntime} += \text{實際運行時間} \times \frac{\text{標準權重}}{\text{任務權重}}$$

    調度邏輯極其簡單：調度器永遠選擇 vruntime 最小 的任務來執行。 - 權重越高 $\rightarrow$ vruntime 增加越慢 $\rightarrow$ 更有機會被選中 $\rightarrow$ 獲得更多 CPU 時間。 - 睡眠太久 $\rightarrow$ vruntime 遠低於系統平均值 $\rightarrow$ 喚醒後獲得「特權」立即執行。

    02. SRE 的噩夢：Weight vs Quota (權重 vs 配額)

    在 K8s 中，我們定義 requests 和 limits。這對應到內核中完全不同的兩種機制，而這正是導致 p99 尖峰的根源。

    1. 相對公平性 (Weight / Requests)

    • 對應機制：cpu.weight (cgroup v2) / cpu.shares (v1)。
    • 行為：這是一種「競爭關係」。如果節點空閒，即使你的 request 很低，你也可以使用 100% 的 CPU。只有當多個 Pod 競爭 CPU 時，權重才決定誰分得多。
    • 影響：導致 Starvation (飢餓)。低權重任務在飽和狀態下會感覺到明顯的延遲。

    2. 絕對限制 (Quota / Limits)

    • 對應機制：cpu.max (v2) / cpu.cfs_quota_us (v1)。
    • 行為：這是一堵「死牆」。例如 limits: 500m 意味著在每 100ms 的周期內，你最多只能跑 50ms。
    • 影響：導致 Throttling (限流)。一旦觸發，即使整個節點 CPU 使用率只有 1%，你的進程也會被內核強制暫停（Sleep），直到下一個周期開始。

    這就是為什麼你會看到「CPU 使用率低，但延遲極高」 $\rightarrow$ 你撞上了 Quota 牆。

    03. 實戰診斷：如何定位 CPU 瓶頸？

    當服務出現延遲時，請按照以下路徑進行排除：

    第一步：確認是否為 Throttling (限流)

    • 信號：檢查 Prometheus 指標 container_cpu_cfs_throttled_seconds_total $\uparrow$ 或查看 cgroup 檔案 cpu.stat 中的 nr_throttled。
    • 結論：如果該值持續增加 $\rightarrow$ Quota 太緊。
    • 對策：增加 limits 或將 QoS 類別改為 Burstable (移除 limit)。

    第二步：確認是否為 Scheduling Latency (調度延遲)

    • 信號：使用 perf sched latency 觀察 max-delay。
    • 結論：如果 vruntime 競爭激烈 $\rightarrow$ CPU 飽和。
    • 對策：水平擴容 (Scale out) 或提升節點規格 (Scale up)。

    第三步：確認是否為 Steal Time (宿主機搶占)

    • 信號：top 指令中的 %st 欄位 $\uparrow$。
    • 結論：虛擬機被 Hypervisor 搶占 $\rightarrow$ 宿主機超賣 (Overcommit)。
    • 對策：將 VM 遷移至低負載宿主機。

    結語：從「猜測」到「量化」

    不要嘗試透過盲目增加 CPU 核心來解決延遲問題。首先區分你是處於 「權重競爭」 還是 「配額限流」。

    對於延遲敏感型服務，最專業的配置是：設定適當的 requests 以確保權重，同時極其謹慎地設定 limits（或在受控環境下移除 limits），將 CPU 的控制權交還給內核的公平調度邏輯。
