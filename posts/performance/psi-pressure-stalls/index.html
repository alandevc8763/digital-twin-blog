    別被 CPU 使用率騙了：用 PSI 捕捉系統資源的「真實痛苦」

    在 SRE 的日常診斷中，最令人沮喪的場景莫過於此：監控面板顯示 CPU 使用率僅 40%，記憶體也綽綽有餘，但應用程式的 p99 延遲卻在瘋狂飆升。

    大多數工程師的第一反應是檢查 CPU Throttling 或網路抖動，但如果這兩者都正常，你可能遇到了 Linux 核心中最隱蔽的性能殺手——資源飢餓 (Resource Starvation)。

    傳統的「利用率 (Utilization)」指標其實是一個巨大的誤導。它衡量的是活動量 (Activity)，而非痛苦程度 (Pain)。一個 CPU 核心即使在 100% 運作，如果它是高效地執行指令，系統是健康的；但如果一個任務因為等待 I/O 或記憶體回收而處於 D 狀態，即使 CPU 利用率極低，你的應用程式依然處於「停擺」狀態。

    為了量化這種「痛苦」，Linux 核心引入了 PSI (Pressure Stall Information)。

    核心邏輯：從「利用率」轉向「壓力值」

    PSI 的核心洞察在於：真正影響用戶體驗的不是資源被用了多少，而是有多少任務因為等待資源而被迫停下來。

    PSI 將資源爭搶分為兩種維度：

    1. some (部分停擺)：至少有一個任務在等待資源，但系統中仍有其他任務在運行。這代表系統開始出現延遲，性能正在退化。
    2. full (完全停擺)：所有非空閒任務在同一時間都被阻塞。這代表資源已完全耗盡，系統進入了短暫的「死鎖」或極端飢餓狀態。

    三大資源的痛苦特徵

    資源 some 的表現 full 的表現 SRE 的直覺診斷
    CPU 尾端延遲 (Tail Latency) 增加 應用程式完全失去響應 (Frozen) CPU 過載 $\to$ CFS Throttling $\to$ 中斷風暴
    Memory 偶發性卡頓，Page Cache 劇烈抖動 系統大面積 Swap，觸發 OOM-Killer 記憶體洩漏 $\to$ 激進的回收 $\to$ 記憶體不足
    I/O 磁碟寫入緩慢，iowait 升高 應用程式阻塞在同步寫入，檔案系統掛起 磁碟 IOPS 飽和 $\to$ EBS 吞吐量達上限 $\to$ 核心死鎖

    SRE 實戰：如何解讀 PSI 訊號？

    你可以直接透過 /proc/pressure/ 下的偽檔案查看目前的壓力狀況：

    cat /proc/pressure/memory
# 輸出示例：some avg10=0.00 avg60=0.05 avg300=0.01

    解讀：在過去 60 秒中，有 5% 的時間至少有一個任務因為記憶體問題而停擺。

    診斷矩陣：利用率 $\times$ PSI

    當你將 PSI 與傳統利用率結合時，真正的瓶頸會立即現形：

    • 高利用率 + 高 PSI $\rightarrow$ 真實過載：資源確實不夠用了。$\text{方案} \to$ 擴容 (Scale-up/out)。
    • 低利用率 + 高 PSI $\rightarrow$ 隱形阻塞：這是最危險的訊號。代表你的應用程式被某個低效的資源路徑（例如單線程阻塞在慢速磁碟 I/O）卡死了。$\text{方案} \to$ 優化 I/O 模式或檢查硬體故障。

    避坑指南：PSI 的陷阱

    1. 不要對 some 發出 P1 告警：some 是預警，full 才是危機。除非 some 持續處於高位，否則它通常只是短暫的尖峰。
    2. 不要忽略 I/O PSI：許多 SRE 習慣監控 CPU 和 Mem，但 I/O PSI 是唯一能區分「程式碼寫得爛」與「核心 Block Layer 太慢」的指標。
    3. 優先看 Cgroup PSI：在容器化環境中，全域的 /proc/pressure 會被其他容器的噪音干擾。請務必檢查 /sys/fs/cgroup/.../cpu.pressure 以精確定位是哪個 Pod 在痛苦。

    總結

    PSI 將 Linux 的可觀測性從「資源佔用」提升到了「任務感知」。下次當你面對「CPU 沒滿但系統很慢」的詭異現象時，請直接打開 /proc/pressure。

    記住：利用率衡量的是機器在做什麼，而 PSI 衡量的是應用程式在承受什麼。
